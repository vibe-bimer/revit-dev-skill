# Revit 风管分段开发流程

## 适用场景

当用户提出以下需求时使用本 skill：
- 风管按指定长度切分
- 连接件插入 / 回退
- 选中范围 vs 系统范围处理
- 三通 / 变径 / 软风管的支持或限制
- 风管分段的 bug 修复、优化、回归验证

## 先冻结的默认策略

在没有用户进一步约束时，优先采用：
- **范围**：默认只处理选中风管
- **尾段**：允许保留短尾段，不强行补成完整段
- **连接件**：按 `FamilySymbol` / 族类型选择
- **不兼容**：单元素跳过并记录原因，不整批失败
- **复杂网络**：v1 默认限制，不主动硬吃三通/变径/软风管
- **验收**：编译通过 + Revit 2026 smoke test 通过

## 工作步骤

### 1) 先查现有实现
重点看这些点：
- 命令入口与对话框默认值
- `Scope` 是否同时支持选中 / 系统范围
- 是否已经有 `BreakCurve`、`NewUnionFitting`、`NewTransitionFitting`、Connector 处理逻辑
- 是否存在“路径账本”与“逐管切分”混用的旧逻辑

### 2) 冻结边界条件
至少确认：
- 处理范围：选中 / 系统
- 尾段策略：保留短尾段还是并入前段
- 最小分段长度：不合法直接拒绝
- 复杂网络：支持还是明确限制
- 失败策略：跳过单个元素还是整体失败

### 3) 写 spec 再改代码
spec 里必须写清：
- 目标
- 变更文件
- API 依据
- 事务边界
- 回滚方式
- 验收标准
- 限制范围

### 4) 编码后立即做两类验证
- **静态验证**：编译能过
- **动态验证**：Revit 2026 内 smoke test

## 常见坑

### 坑 0：口径漂移比算法错误更常见
风管分段这类需求经常不是“算法全错”，而是**代码、对话框、summary、截图**四层口径不一致。遇到用户拿截图对比图3时，必须先确认：
- 代码实际执行的是主路径、全系统逐根分段，还是账本连续排布
- 对话框文案有没有还在说旧模式
- summary 有没有把旧模式术语继续露出来
- 截图里看到的结果是不是和弹窗数值一致

如果用户明确要的是“图3效果”，不要只修账面，要把**真实几何结果 + 弹窗文案 + 截图观感**一起收口，否则会出现“算对了但图不对”的返工。

### 坑 0.1：只改文案不会改变截图
如果用户说“执行完后跟刚才一样”，优先怀疑不是 UI 文案，而是**真实几何流程没变**：
- 仍然走旧的主路径/账本逻辑
- 支路虽然被收集了，但没有进入真正的切分执行器
- 段长校正只是按末端偏差回拉，导致 1200mm 变成 1203mm 之类的视觉漂移

这时要直接追到 `DuctSegmentationExecutor` / `DuctSegmentationGeometryHelper` / `DuctPathPlanner`，不要在 summary 上打补丁糊弄。

### 坑 0.2：1200mm 变 1203mm 多半是“终点回拉”方式错了
如果分段后截图里看起来总是多出几毫米，常见根因不是换算公式本身，而是：
- 以末端偏差做回拉，偏差会被上一轮补偿残留放大
- 连接件/补偿值与切断点叠加，结果被校正到“接近正确但不精确”

更稳的做法是：**固定起点，直接把终点拉到目标长度**，让每段长度由目标值重建，而不是从偏差倒推。

### 坑 0.3：EntireSystem 的标签改了，但行为还在主路径
当用户目标是“主干+支路都像工程图一样被切出来”时，单改 `summary`、`dialog`、`scope` 默认值不够；必须确认：
- `CollectDuctsInSystem` 收集到的 ducts 是否真的进入执行器
- `SegmentDuctInSubTx` 是否对每根都实际执行
- summary 是否把旧的“主路径连续排布”术语完全清掉

否则最终会出现“口径看上去已经全系统了，图上还是老样子”的假完成。

### 坑 0.4：不要用“末端回拉/重写曲线”去凑整段长度
如果截图里短支路被硬拽成 1200 的整数倍，通常不是切点算错，而是长度校正器在**改写几何**：
- 以末端偏差回拉，容易把偏差残留放大
- 对短支路/末端枝杈尤其危险，会把真实几何压扁

更稳的做法是：**保留当前真实长度，只把偏差写入补偿项，下一刀再处理**。校正器的职责是记录偏差，不是把模型强行修成“看起来整齐”的假结果。

### 坑 0.5：`--build-only` 只编译，不同步 Windows 真正运行的那份代码
Revit 插件验收时，`./build.sh --build-only` 经常会制造“本地改了、截图还是旧的”的错觉，因为它只做编译，不会把新代码 push/pull 到 Windows 端再部署。要验证截图和 Revit 里实际效果，必须走完整的 `./build.sh` 同步链路。

### 坑 1：EntireSystem 的统计口径被旧变量覆盖
在主路径模式里，`skippedCount` 不能只写成支路未处理数；它通常需要保留**预校验失败 + 支路未处理**两部分。映射路径级统计到旧 summary 变量时，优先做累加而不是赋值覆盖，否则 UI 会出现“前面算过的 skip 被后面冲掉”的假象。

### 坑 1：目标是“图3式工程表达”时，不要死守主路径账本
如果用户要的是**主干+支路都连续铺开、视觉上像工程图**，而不是“只把最长主路径算对”，就不要把 EntireSystem 固定绑死在主路径 ledger 上。实际更稳的做法是：
- EntireSystem 走全系统逐根分段，主干和支路统一使用同一套分段/标注逻辑；
- summary 允许在没有 ledger 时退回到“全系统结果”口径；
- 账本模型仍可保留给主路径连续排布类需求，但不要误用到所有图像目标上。

### 坑 2：远端 build.sh 失败不一定是代码坏了
`./build.sh --build-only` 如果报 `MSB3231` / `Access denied`，常见原因不是编译错误，而是 Windows 端 Revit.exe 正在运行，导致 `Addins\\2026\\SuperRoky` 下 DLL 被锁定。先关掉 Revit.exe，再重跑 build；否则会把部署锁定误判成代码回归。

## 常见坑

### 坑 1：把“能切出 1 段 + 短尾”误判成不可分段
正确条件不是“至少 2 段才进入”，而是：
- 只要总长能切出一个完整段，就应该允许进入流程
- 只有总长 ≤ 分段长度 + 容差时，才跳过

### 坑 1：系统级路径账本和逐管严格切分混在一起
如果同时有：
- 主路径连续排布
- SelectedOnly 单管严格切分

要先明确谁是主路径，谁是 v1 范围。否则会出现逻辑互相打架。

### 坑 1.1：`EntireSystem` 不等于“把主路径改名叫全系统”
这次的实际教训是：如果用户看到的症状是**主干切了、支路没动**，那通常不是 summary 文案问题，而是路径构建层本身还在用 **`BuildMainPath` / 贪心 DFS**。它会天然丢掉支路，只保留最长可达路径。

当用户明确想要“全系统都继续切”时，必须先确认是不是需要：
- 只处理一条连续主干；还是
- 把系统拆成多个连续路径，逐路径处理

如果需求是后者，不能继续把 `EntireSystem` 绑定在单条主路径上，否则会出现“账面全对、图上没切支路”的假完成。

#### 解决方案模式：`BuildSystemPaths()` 多路径分解

当遇到上述问题时，不要继续在 `BuildMainPath` 上修修补补。直接在 `DuctPathPlanner` 中新增 `BuildSystemPaths(...)` 方法，核心思路：

1. **入口**：调用 `CollectDuctsInSystem(doc, selectedDuct)` 获得全系统 duct 集合（已去过重和 Fitting 过滤）。
2. **路径拆解**：
   - 先用 `BuildMainPath(...)` 从入口 duct 提取第一条连续路径，该路径上的 duct 标记为已用。
   - 遍历剩余 duct，每找到一个未用的 duct，用 `GetConnectedDuctSegments(...)` 从它出发 BFS 收集连续子路径（只取 duct，跳过已用的）。
   - 每个这样的连续片段构成一条独立路径，存为 `List<List<Duct>>`。
3. **输出**：返回 `(MainPath, BranchPaths)` 结构。
4. **执行端**（`DuctSegmentCommand`）：
   - 先处理主路径（全链路账本 + `ExecuteLedgerBreaksOnMainPath`）。
   - 再对每条支路路径，独立走一遍：建账 → 子事务 → 切分 → 提交/回滚。
   - 统计统一汇总（主路径 + 各支路的结果累加）。
5. **关键坑**：
   - 支路路径独立建账时，`BuildPathLedger` 的参数是支路总长度，不继承主路径的偏移量。
   - 每条支路内的切点执行用 `SegmentDuctInSubTx`（单根严格分段），不是主路径的 `ExecuteLedgerBreaksOnMainPath`。
   - 路径种子不要重复使用已被主路径或之前支路标记过的 duct。

### 坑 3：复杂网络边界不清
三通、变径、软风管、占位线等，如果没有明确规则，先限制，不要硬写“或明确说明限制”但实际没定义。

### 坑 4：直接 `dotnet build` 裸跑项目
这类 Revit 项目常依赖 solution configuration / RevitVersion / build script。
优先使用仓库自带脚本，例如：
- `./build.sh --build-only`

不要先假设 csproj 可以直接裸编译。

### 坑 5：EntireSystem 的统计口径容易互相覆盖
路径级统计里，`skippedCount` 不能被主路径支路统计覆盖，必须把预校验跳过和支路未处理累加；`RemainderSegments` 也要按账本余量真实推导，不能默认空值或写死 0。

### 坑 6：账本切点要优先按 offset 映射，不要继续用“起点 + 段长 + 半个连接件”的滚动偏移
主路径连续排布的核心是：先生成账本，再把 `BreakOffsetsFt` 映射到 station 和局部曲线；`connectorHalfFt` / `_lengthCompensationFt` 这类滚动补偿只适合旧的 SelectedOnly 逻辑，不该成为 EntireSystem 主干。

## 验证建议

### 编译
优先使用仓库 build 脚本。

### 回归检查
至少确认：
- 命令按钮还能显示
- 选中风管能进入分段流程
- 短尾段符合预期
- 复杂网络限制符合说明
- 失败原因能读懂，不是空报错

### Smoke test
在 Revit 里验证：
- 执行命令
- 选中目标风管
- 分段结果与 Criteria 一致
- UI 未被破坏
